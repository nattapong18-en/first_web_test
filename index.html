ภาษา C: รากฐานที่ยังค้ำระบบคอมพิวเตอร์ทั้งโลก

จากเครื่อง PDP-11 เครื่องเดียวใน Bell Labs สู่ภาษาที่อยู่เบื้องหลัง Unix, Linux kernel, และเฟิร์มแวร์ในไมโครคอนโทรลเลอร์ทุกตัวที่คุณเคยแตะ — นี่คือเส้นทางกว่า 50 ปีของ C

ผู้สร้าง — Dennis Ritchie สถานที่ — Bell Labs, Murray Hill, NJ มาตรฐานล่าสุด — ISO/IEC 9899:2023 (C23)
01

ก่อนจะมี C

C ไม่ได้เกิดขึ้นลอย ๆ — มันคือคำตอบต่อปัญหาจริงของทีมที่กำลังสร้างระบบปฏิบัติการ

ปี 1969 Ken Thompson และ Dennis Ritchie ที่ Bell Labs เริ่มพัฒนา Unix บนเครื่อง PDP-7 ที่แทบไม่มีใครใช้แล้ว โดยเขียนด้วยภาษาแอสเซมบลีทั้งหมด ซึ่งหมายความว่าโค้ดทุกบรรทัดผูกติดกับสถาปัตยกรรมของเครื่องนั้นเครื่องเดียว ย้ายไปเครื่องอื่นแทบเป็นไปไม่ได้

Thompson จึงพัฒนาภาษาชื่อ B ขึ้นมาก่อน โดยดัดแปลงมาจากภาษา BCPL อีกที B เป็นภาษาแบบ typeless เรียบง่าย เหมาะกับการเขียนเครื่องมือระบบ แต่ยังมีข้อจำกัดสำคัญ: มันจัดการกับข้อมูลเป็น "word" เดียวเท่านั้น ไม่มีชนิดข้อมูลแยกอย่าง char หรือ struct ทำให้ไม่เหมาะกับฮาร์ดแวร์รุ่นใหม่อย่าง PDP-11 ที่เริ่มมีการอ้างอิงข้อมูลระดับไบต์

02

กำเนิด C และการเขียน Unix ใหม่

1972

Dennis Ritchie สร้างภาษา C

บนเครื่อง PDP-11 Ritchie พัฒนา C ต่อยอดจาก B โดยเพิ่มระบบชนิดข้อมูล (type system) เช่น char, int, และที่สำคัญที่สุดคือแนวคิด struct กับตัวชี้ (pointer) ที่ผูกกับชนิดข้อมูลอย่างชัดเจน ทำให้เขียนโค้ดที่แม็ปกับหน่วยความจำจริงได้แม่นยำ แต่ยังคงเป็นนามธรรมพอที่จะพอร์ตข้ามเครื่องได้

1973

Unix kernel ถูกเขียนใหม่ด้วย C

นี่คือจุดเปลี่ยนที่แท้จริง: แทนที่จะผูก Unix ไว้กับแอสเซมบลีของ PDP-11 ทีมงานเขียน kernel ใหม่เกือบทั้งหมดด้วย C ผลลัพธ์คือ Unix กลายเป็นระบบปฏิบัติการตัวแรก ๆ ที่ "พอร์ตได้" (portable) — ย้ายไปรันบนสถาปัตยกรรมอื่นได้โดยแก้โค้ดเพียงส่วนน้อย นี่คือจุดที่ C กับ Unix ผูกชะตากันตลอดกาล

1978

หนังสือ "The C Programming Language"

Brian Kernighan และ Dennis Ritchie ตีพิมพ์หนังสือที่รู้จักกันในชื่อ "K&R" ซึ่งทำหน้าที่เป็นทั้งคู่มือภาษาและมาตรฐานอย่างไม่เป็นทางการนานกว่าทศวรรษ ก่อนที่จะมีมาตรฐานอย่างเป็นทางการ นักพัฒนาทั่วโลกอ้างอิงพฤติกรรมของภาษาจากหนังสือเล่มนี้เพียงเล่มเดียว

03

การกลายเป็นมาตรฐาน

1989 / 1990

ANSI C (C89) และ ISO C90

เมื่อ C ถูกใช้แพร่หลายและมีหลายสำนักพัฒนาคอมไพเลอร์ต่างค่ายกันไปตามใจ ANSI จึงตั้งคณะกรรมการกำหนดมาตรฐานอย่างเป็นทางการ ผลลัพธ์คือ C89 ซึ่ง ISO รับรองต่อในชื่อ C90 นี่คือจุดที่ function prototype, void, และไลบรารีมาตรฐานอย่าง stdlib.h กลายเป็นส่วนหนึ่งของภาษาอย่างเป็นทางการ

1999

C99

เพิ่มความสามารถที่โปรแกรมเมอร์รุ่นหลังคุ้นเคย เช่น คอมเมนต์ //, การประกาศตัวแปรกลางฟังก์ชันได้ (ไม่ต้องอยู่บนสุดของ block), inline function, variable-length array, และชนิดข้อมูลที่ระบุขนาดชัดเจนอย่าง uint32_t ใน stdint.h — สิ่งที่งานเขียนเฟิร์มแวร์สมัยนี้ยังพึ่งพาอยู่ทุกวัน

2011

C11

ตอบรับยุคมัลติคอร์อย่างเป็นทางการ ด้วย threads.h สำหรับจัดการเธรดในตัวภาษา, stdatomic.h สำหรับ atomic operation, และ _Generic สำหรับเขียนโค้ดที่ทำงานต่างกันตามชนิดข้อมูล (type-generic) โดยไม่ต้องพึ่ง preprocessor macro ล้วน ๆ

2017 / 2023

C17 และ C23

C17 เป็นการแก้บั๊กของข้อกำหนดใน C11 โดยไม่เพิ่มฟีเจอร์ใหม่ ส่วน C23 (ISO/IEC 9899:2024) เป็นมาตรฐานล่าสุด นำ bool, true/false เข้ามาเป็นคีย์เวิร์ดโดยตรง (ไม่ต้อง include stdbool.h อีกต่อไป) พร้อมปรับปรุงเรื่อง attribute และการอนุมานชนิดข้อมูลให้ทันสมัยขึ้น

04

ทำไม C ยังสำคัญ

Systems

รากฐานของระบบปฏิบัติการ

Linux kernel, Windows NT kernel บางส่วน, และ Unix-like ทุกตัวยังเขียนด้วย C เป็นแกนหลัก เพราะให้การควบคุมหน่วยความจำและฮาร์ดแวร์แบบตรงไปตรงมาโดยไม่มี runtime แทรกกลาง

Lineage

ต้นแบบของภาษาต่อมา

C++ เริ่มจาก "C with Classes", Java และ C# หยิบไวยากรณ์ curly-brace มาโดยตรง แม้แต่ Rust เองก็ยังต้องอ้างอิงโมเดลหน่วยความจำของ C เป็นจุดเทียบเวลาออกแบบระบบ ownership

Embedded

ภาษาเริ่มต้นของไมโครคอนโทรลเลอร์

ตราบใดที่ยังมีชิปที่ไม่มีทรัพยากรพอสำหรับ runtime หนัก ๆ C ยังเป็นตัวเลือกอันดับแรกของ toolchain อย่าง ESP-IDF, STM32 HAL หรือ AVR-GCC เพราะแม็ปเข้ากับ register และหน่วยความจำได้ตรงที่สุด

Philosophy

"Trust the programmer"

C ไม่กัน null pointer, ไม่กัน buffer overflow ให้อัตโนมัติ — เป็นปรัชญาที่ยกภาระความปลอดภัยของหน่วยความจำให้ผู้เขียนโค้ดทั้งหมด ข้อถกเถียงเรื่องนี้เองที่ผลักดันให้เกิดภาษาอย่าง Rust ในเวลาต่อมา

05

บันทึกส่งท้าย

ARCHIVED

กว่า 50 ปีหลัง PDP-11 เครื่องนั้น C ยังไม่ตาย และไม่มีทีท่าจะตาย เพราะปัญหาที่มันแก้ในปี 1972 — การเขียนโค้ดที่ควบคุมฮาร์ดแวร์ได้ตรงจุด แต่ยังพอพอร์ตข้ามเครื่องได้ — ยังเป็นปัญหาเดียวกับที่งานเฟิร์มแวร์บน ESP32 หรือ MCU ตัวไหนก็ตามต้องเจอในวันนี้
